# AgentMesh Internal Distribution Shape

`schema_version`: 1

This document defines the internal team distribution boundary for the monorepo
package split. It is a release gate, not a marketing package matrix.

## CLI Tarball

The root package `@jinhx128/agentmesh` is the only installable CLI tarball.
Workspace packages under `packages/*` and `apps/*` stay `private: true`; they
are build units and source ownership boundaries, not independently published
npm artifacts.

The root `bin.agentmesh` points to the built executable:

```text
dist-node/packages/cli/src/cli.js
```

The root tarball whitelist is intentionally narrow:

- `dist-node/packages/`
- `dist-node/apps/studio/`
- `dist-node/apps/studio-web/`
- `packages/skills/agentmesh-skill/`

This keeps the install surface focused on the executable CLI, runtime code,
read-only SDK code, Studio app-server/frontend build output, skill template,
and declaration files. Source TypeScript, tests, changelog entries, review
records, and desktop packaging internals are not part of the CLI tarball.

## Dependency Boundary

The root package has no `@agentmesh/*` workspace dependencies. Cross-package
source imports are rewritten in built JavaScript before packing, so the
installed CLI has no unresolved private-workspace dependency gap.

Runtime dependencies that are not internal workspace packages remain normal
root package dependencies and are installed by npm when the tarball is
installed.

## Types

The root TypeScript build emits declaration files into `dist-node`. The CLI
tarball must include declaration files for the built internal runtime surface,
including the CLI entrypoint, core schemas, runtime APIs, read-only SDK APIs,
skill install/verify APIs, and Studio app-server boundary.

## Verification

Required gates:

- `npm run build`
- `npm test`
- `npm pack --dry-run`
- clean install smoke in a temporary project
- `npm run cli:install-smoke`

The clean install smoke installs the root tarball, then executes:

```sh
agentmesh --help
agentmesh doctor --json
agentmesh skill show
agentmesh skill install --target codex --force
agentmesh skill verify --target codex --json
```

The smoke uses an isolated temporary home and a minimal project config so it
validates the installed package, not a developer's personal provider login
state.

## Command Install Modes

Users and internal teammates can make `agentmesh` PATH-visible through:

- source checkout plus `npm link`
- local tarball generated by `npm pack`
- public npm registry install of `@jinhx128/agentmesh`
- internal/private registry install of the same root tarball, if mirrored

The root package is publishable with `private: false` and
`publishConfig.access = "public"`. Any registry path must preserve the same
tarball boundary and clean install smoke as the local tarball.

Release owners publish the public npm package through:

```sh
npm run publish:npm
```

If npm prompts for two-factor authentication, pass the one-time password with:

```sh
npm run publish:npm -- --otp <one-time-code>
```

Before installing or updating from Desktop, the app detects the PATH-visible
`agentmesh`, executes `--version`, and shows the path and actual installed version.
It invokes public npm without a bin path input, then re-detects the command. The expected
verification after any command install is:

```sh
agentmesh --help
agentmesh doctor --json
agentmesh skill verify --target <host> --json
```

Permission, network, npm discovery, and PATH-order failures must remain visible
to the user.

See `docs/distribution/cli-command-install.md` for the source checkout,
tarball, npm registry, and PATH-visible `agentmesh` user flow.

## Desktop Channel

The Desktop channel does not consume the npm CLI tarball for Studio runtime.
`AgentMesh.app` bundles its own sidecar input from app-server/runtime/studio-web
build artifacts:

- built Node runtime and package code under `dist-node`
- built Studio web assets under `dist-node/apps/studio-web/frontend`
- app-managed runtime dependencies under the desktop sidecar resources
- built Skill resources used by the per-target Skill installer

Desktop packaging consumes the bundled App Server and runtime APIs for Studio
reads and writes. Studio behavior must not depend on a PATH-visible,
npm-installed, or app-bundled CLI command artifact.

DMG-only installs satisfy Desktop Studio only. Entry agents and installed
Skills require a user-selected PATH-visible `agentmesh`. The Settings /
Agent Integrations command-line action is optional, installs or updates the
public npm package, and verifies the PATH-visible result without a bin path input.
