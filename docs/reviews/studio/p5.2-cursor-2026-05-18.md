## Should Fix

1. **`packages/app-server/src/integrations.ts` (and/or `packages/skills/src/verify.ts`) — CLI status vs actual install path**  
   `requires_confirmation` is only `pathCommand.found && pathCommand.path !== targetPath` (see `commandLineToolReport`). It does **not** cover “file already exists at the chosen `bin_dir` with different content,” and the GET report always uses `default_bin_dir` for `target_path`, while the UI can send a different `bin_dir`. The POST still correctly 409s, but the product asks to **show** status before requiring confirmation; users can see `requires_confirmation: false` and still hit 409, or not understand why. **Tighten** by extending `requires_confirmation` to include target-path replace cases (for the effective directory), adding a preview/status call that takes `bin_dir`, or making the UI conservative (e.g. always require the checkbox when the effective directory differs from default or when `target_path` might already exist).

2. **`apps/studio-web/src/features/settings/AgentIntegrationsPanel.tsx` — Confirmation UX**  
   The API exposes `command_line_tool.requires_confirmation`, but the panel never surfaces it; the checkbox label only says “Confirm replacement,” not **PATH shadowing / another `agentmesh` on PATH** (per `docs/distribution/cli-command-install.md`). Align copy and UX with `path_command` + `requires_confirmation` so users know why confirmation is required.

3. **Same file — skill install result copy**  
   `resultText` for skills only reports how many targets succeeded (`installed_targets.filter(ok)`). Partial failures can look like full success. Prefer including failed targets or an explicit “some targets failed” line when any `ok === false`.

4. **`packages/skills/src/verify.ts` — `agentmeshSkillMarkdown(cwd)` first candidate**  
   The first lookup is `cwd/packages/skills/agentmesh-skill/SKILL.md`. Opening the AgentMesh repo (or a tree that contains that path) as the Studio workspace makes Desktop installs use **workspace** skill text instead of the app-bundled artifact, which weakens “package-resource consistency” for P5.2. **Consider** preferring `moduleProjectRoot()` / bundle-root skill when installing from Desktop, or gating the `cwd` candidate behind a clear policy.

5. **`docs/distribution/*.md` (or desktop doc) — Wrapper durability**  
   The wrapper bakes **absolute** `nodePath` and `cliPath` at install time (`commandLineToolWrapper`). Document that App moves, updates that change `.app` path, or reinstalls can break an existing wrapper so power users know to re-run install.

## Nit

1. **`packages/app-server/src/integrations.ts` — `isExecutableFile`**  
   Only checks `S_IXUSR`. Unusual permission bits could make a PATH-visible binary not detected; low risk on macOS for typical `0755` installs.

2. **`apps/studio-web/src/features/settings/AgentIntegrationsPanel.tsx`**  
   Skill rows show `status` but not `expected_path` / hints from the report; debugging installs is harder than necessary.

3. **Shared skill targets**  
   Codex/Cursor/Gemini/OpenCode/Copilot all map to the same `.agents/skills/agentmesh/SKILL.md`; selecting several is mostly redundant (behavior is consistent with `installSkill` + tests, but the grid can over imply “separate files per host”).

## Summary

**P3 boundary / mutations:** Studio mutations still go through `runStudioMutation` → runtime `dispatchFlowStage` / `attachStageArtifact` with `entrypoint` passed in-process; they do not shell out to a global `agentmesh` CLI. Integrations are separate HTTP routes and optional. That matches “Desktop Studio mutations use app-bundled App Server/runtime APIs and must not call CLI.”

**P5.2 DMG / packaging:** `tauri.conf.json` is **DMG-only**, `runtime.npm_cli_shared_install` is false in the manifest, smoke checks bundle `dist-node/packages/cli`, `dist-node/packages/skills`, and `packages/skills/agentmesh-skill`, and **rejects** shipping a blanket `dist-node` tree—good for resource hygiene. The sidecar launcher checks still forbid PATH `node` / `agentmesh` for the managed runtime.

**Install safety (CLI):** Conflict detection requires `confirm_existing: true` when another executable wins PATH first or the install path exists with different content; tests cover 409, install, and “PATH still points at the old binary after install to another dir,” which matches the honest semantics of `path_command` after install.

**Skills:** Targets codex/cursor/gemini/opencode/copilot share `.agents/skills/agentmesh/SKILL.md`; claude uses `.claude/skills/agentmesh/SKILL.md`, aligned with your stated layout. POST parses an array of supported targets and returns per-target results.
