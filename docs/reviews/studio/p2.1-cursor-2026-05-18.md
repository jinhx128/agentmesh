## Non-blocking notes

1. **`StudioMutationOptions.cliPath`** is still threaded through `server.ts` but **`mutations.ts` does not use `cliPath`** for execution anymore—only the synthetic `command` array for UI/telemetry. Consider dropping it from mutation options or documenting it as UI-only to avoid implying CLI execution.
2. **Synthetic `command` arrays** now look like `runtime flow …` / `runtime agents …` (not `node … agentmesh …`). Tests still pass because assertions are substring-based; the UI may show a **“runtime …” prefix**, which is accurate but slightly different from a copy-paste CLI invocation (`agentmesh …`). Optional follow-up: align copy or labels in the manual / UI.
3. **Embedded manual copy** in `packages/app-server/src/assets.ts` still describes the **CLI as the write channel** in at least one i18n string. That remains broadly true for terminal use but is **easy to misread** next to an in-process App Server; consider wording that distinguishes **terminal CLI** vs **Studio/App Server (bundled runtime APIs)** without a big rewrite.
4. **CLI consistency (pre-existing):** `packages/cli/src/cli.ts` passes `configPath` into `agentsAdd` but not into `agentsRemove` / `agentsEnable` / `agentsDisable`. If `--config` is meant to affect all agent writes uniformly, that gap predates this extraction but is worth a separate fix if it bites real usage.

## Residual risks

- **Concurrency:** `appendCallAdoptionEvent` uses atomic record writes (same as CLI); there is no separate run-directory lock for call adoption—if that ever races with another writer on the same call dir, behavior depends on filesystem ordering (same as before this change).
- **Scope defaults:** Studio agent create still defaults **omitted `scope` to `project`** (`safeScope` in `agent-lifecycle.ts`), while CLI `agents add` defaults omitted `--scope` to **user** (`agentWriteScope`). That asymmetry is long-standing for Studio vs terminal; worth keeping in onboarding/docs so users are not surprised.
