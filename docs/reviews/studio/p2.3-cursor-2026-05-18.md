# agentmesh-review-markdown-v1

## Blocking findings

1. **`catalog.ts` is allowlisted as a whole file, not “MCP read-only diagnostics” only.**  
   `allowedAppServerCliDiagnosticFiles` exempts **`packages/app-server/src/catalog.ts`** from *all* of the forbidden patterns (`node:child_process`, `@agentmesh/cli`, `packages/cli`, `spawn*`/`spawnSync`, `process.execPath`). Any future change that adds **`execSync`**/**`execFileSync`**/**`fork`** (not covered anyway — see Should-fix #1) would still bypass if placed in another file until you extend rules; worse, **`spawn`/CLI orchestration moved into `catalog.ts` would never trip this gate**, even for non-MCP or write-ish behavior. For a strict “release gate,” that carve-out is **materially broader than the stated MCP-only carve-out.**

## Should-fix items

1. **`child_process` bypass via APIs other than `spawn`/`spawnSync`.**  
   The gate uses `\bspawn(?:Sync)?\b` plus `node:child_process` / `from 'child_process'`. A maintainer could call **`execSync`/`execFile`/`execFileSync`/`fork`** (still `node:child_process`) **without mentioning `spawn`**, and **`check-boundaries.mjs` and the analogous test regexp would not flag it** (`package-structure.test.ts` mirrors the same gap). Worth aligning with intended policy (“no subprocess CLI for write paths”) or explicitly documenting that **only spawn-shaped calls are guarded**.

2. **Test vs script symmetry on `@agentmesh/cli`/`packages/cli`.**  
   The script forbids **`@agentmesh/cli` OR `packages/cli`** under `packages/app-server/src/` (excluding `catalog.ts`). Tests split this into a separate **`@agentmesh/cli` import-only** loop plus a **`packages/cli` substring** rule in the spawn-related regex. Functional overlap is OK today, but **drift risk**: someone could widen/narrow only one layer. Prefer **one canonical pattern source** or mirror both tokens in **both assertions**.

3. **Heuristic regexp vs human text / literals.**  
   `\bspawn(?:Sync)?\b` applies to raw file text, so incidental English like the word **`spawn`** in user-facing strings (i18n) could trip CI even when no code paths exist. Rare, but brittle; consider comment/string handling later if noisy.

## App Server write path vs CLI/spawn (fact check)

| Area | CLI / `child_process` for writes? |
|------|-----------------------------------|
| **`packages/app-server/src` outside `catalog.ts`** | Repo search finds **no** `node:child_process` / `spawn` / matching `exec*`-as-child-process usage; **`mutations.ts` uses `@agentmesh/runtime`** (`dispatchFlowStage`, `retryFlowStage`, `resumeFlow`, `attachStageArtifact`) — **does not spawn the CLI.** |
| **`catalog.ts`** | **`spawnSync` + `process.execPath`** drives **`agentmesh … mcp list --json`** only for MCP listing/diagnostics**; agents/workflows use **`@agentmesh/sdk`**. Fits the intentional exception narrative, but enforcement is **file-wide**, not function-scoped. |
| **`apps/studio-desktop` TS (“Desktop”)** | **No** Node `spawn`/`child_process` in `src/**/*.ts`; **`process.execPath`** appears in **`sidecar-bundle.ts`** to **pick/copy the bundled Node binary** — build/pack shape, **not App Server mutation I/O.** |
| **Non-TS desktop** | **Tauri Rust** uses `Command::spawn` — **outside** these checks; unrelated to app-server `.mjs`/`.test.ts` gate. |

So: **for the scoped App Server TS package**, the **effective** posture is **“no subprocess CLI on write handlers except MCP list living in `catalog.ts`”;** residual risk is **`catalog.ts`-level exemption breadth** and **non-spawn `child_process` APIs**.

## Allowlist: too broad?

**Yes**, if “too broad” means **narrower than the policy text**:

- Allowed: **anything in `catalog.ts`** that matches the forbidden tokens.
- Stated intent: **read-only MCP diagnostics only.**  
**Mitigation directions** (for later; not patching per reviewer role): narrower allowlist (e.g. only `runCli`/`readMcpList` spans), **`eslint-disable`/pragma with required justification**, or a **tiny `mcp-diagnostics.ts`** as the sole exempt file instead of **`catalog.ts`**.

## Verdict

**Approve with reservations** for P2.3 as an App Server-focused gate: behavior matches **“no CLI spawn on write surfaces; MCP list diagnostics isolated,”** backed by **`mutations` → runtime** and **grep-only CLI usage under `catalog.ts`.** Reservations: **whole-file carve-out + incomplete `child_process` API coverage** are the main **correctness/evasion** worries for calling this gate “complete”; address before treating it as a **bulletproof** release gate.
