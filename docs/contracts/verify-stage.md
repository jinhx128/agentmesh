# Verify Stage Contract

`schema_version`: 1

`verify` is the stage type for producing verification evidence: commands,
tests, builds, smoke checks, regression logs, skipped checks, and residual
verification risk. It does not replace review, and it does not decide release
readiness.

## Stage Semantics

The recommended workflow order is:

```text
plan -> execute -> verify -> review -> decide
```

The built-in Verified Delivery workflow (`w-1ab330ed`) uses this order as the reference
runtime surface for verify-stage packets.

Review consumes verification evidence; it does not substitute for verification.
Decide and release-check remain responsible for final engineering decisions and
release verdict state. A `verify` stage must not write a release verdict, accept
or reject review findings, or update decision state.

## Canonical Artifact

single-agent verify dispatch writes one canonical markdown artifact:

- first `verify` node: `verification.md`
- repeated `verify` nodes: `verification_2.md`, `verification_3.md`, and so on

The artifact id for the first occurrence is `verification`. Later occurrences
use the semantic artifact id `verification_2`, `verification_3`, and so on.

`verification.md` should record:

- commands run
- test/build/smoke/regression logs or summaries
- skipped checks with reasons
- flaky or inconclusive checks
- residual verification risk

The live workspace state is not verification evidence by itself. Evidence must
be written into `verification.md` or another packet artifact with provenance.

## Prompt Assembly

The default `verify` prompt includes:

- Request
- Assignment
- Context
- ordered prior evidence such as `Prior Output: plan (Current Plan)`
  and `Prior Output: execute (Handoff)`

If `verify` appears after another evidence-producing stage, prompt assembly
uses the ordered prior evidence mechanism and includes prior artifacts in
workflow order. If `verify` appears after `review`, it may include prior review
artifacts, but the verify agent still writes verification evidence rather than
classifying findings.

Prompt text must make clear that commands, logs, skipped checks, and risk belong
in `verification.md`. It must not treat live working tree inspection, private
chat state, or unstored terminal output as durable verification evidence.

## Fanout

Multi-agent `verify` dispatch writes isolated per-agent evidence slots before
the runtime writes the canonical aggregate verification artifact.

Per-agent outputs use:

- `outputs/<node-id>/<agent>.md`
- artifact id `output_<node-id>_<agent>`
- artifact kind `stage-output`

The canonical artifact remains the stage artifact:

- first `verify` node: `verification.md`
- repeated `verify` nodes: `verification_2.md`, `verification_3.md`, and so on

`verification.md` is generated by AgentMesh from every assigned verifier result.
It must include a per-agent summary and the available evidence for completed,
reused, failed, or missing verifier outputs. Failed verifier evidence is partial
evidence, not a release verdict, and downstream review/decide stages must treat
it as verification risk until the workflow resolves or retries it.

Fanout retry reuses completed verifier outputs when the expected output file and
artifact manifest entry are both present. It reruns only missing or failed
verifier slots and rewrites the aggregate verification artifact after all
verifiers settle.

## Capability And Compatibility

The `verify capability` is part of the default full-capability agent profile.
`agents add` defaults new agents to the full stage capability set:
`plan`, `execute`, `verify`, `review`, and `decide`. Existing configured agents
are not mutated automatically. A local agent with an explicit `capabilities`
list must include `verify` before worker dispatch can run it for a verify stage.

`agents add` and config documentation must explain that older agents can keep
working for `plan`, `execute`, `review`, and `decide`; they need a capability
update only if they should run `verify`. Explicit capability lists remain
authoritative for role-scoped agents.

`workflow_defaults` and the capability profile schema accept `verify` as a
stage type. Project capability profiles can declare a `verify` stage and
required capability list.

An old packet without `verify` remains readable. A new workflow that includes
`verify` must be validated by the current schema and must produce stage nodes,
assignments, artifacts, timing, and events using the same node-id rules as other
stage types.

`verify` does not produce or mutate `release verdict` fields. Release verdicts
remain derived from `decision.md` and release-check policy evidence.
