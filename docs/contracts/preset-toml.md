# preset.toml Contract

`schema_version`: 1

A preset is a named run recipe that references one workflow and resolves
workflow-specific assignments, defaults, failure policy, fallback routing, and
timeouts before packet creation. Presets live in the preset namespace; a bare
`agentmesh run <id>` resolves only presets. Workflow runs must use
`--workflow <workflow-id>` or `--workflow-file <path>`.

## Registry CLI

```bash
agentmesh preset list
agentmesh preset show <preset-id>
agentmesh preset init --workflow <workflow-id>
agentmesh preset add <preset-file>
agentmesh preset remove <preset-id>
agentmesh preset doctor <preset-id>
agentmesh run <preset-id> --task "..."
```

Rules:

- The preset registry is user-level global at
  `~/.config/agentmesh/presets/*.toml`.
- `preset init` renders a TOML template with derived stage node ids plus
  assignment, failure policy, and fallback placeholders.
- `preset add` validates the referenced workflow, derived node ids, primary
  agents, fallback agents, failure policy shape, and fallback execution bounds
  before copying the file into the registry.
- A bare `agentmesh run <id>` resolves only the preset namespace. Direct
  workflow runs must use `agentmesh run --workflow <workflow-id>` or
  `agentmesh flow run --workflow <workflow-id>`.

## Task Input

Preset runs require exactly one request source:

```bash
agentmesh run release-duo --task "prepare release evidence"
agentmesh run release-duo --task-file ./request.md
```

`--task` and `--task-file` are mutually exclusive. `--task` supplies inline
request text; `--task-file` reads request text from the given file before packet
creation. If both are present, the CLI fails and does not create the packet. If
neither is present, the CLI prints preset run usage.

Preset runs do not accept role flags such as `--plan`, `--execute`, `--verify`,
`--review`, or `--decide`; those are direct workflow run inputs. Preset
assignments come from the preset file plus preset/global defaults.

## Minimal Shape

```toml
schema_version = 1
name = "Release Duo"
workflow = "w-67ef1b1f"

[stage_assignments]
review = ["claude", "antigravity"]
decide = ["current"]
```

Rules:

- Preset ID is generated by `preset add` and stored as the registry filename;
  preset TOML does not carry a user-authored `id` field.
- `workflow` is required and must resolve through the workflow registry unless
  the caller explicitly supplies a temporary workflow file.
- `stage_assignments` keys are stage node ids, not stage types. Repeated nodes
  use ids like `review_2` and `decide_2`.
- Preset assignment may use `current` only as a pure node assignment.
- Every resolved node must have at least one primary assignment after explicit
  node assignment, preset defaults, and global defaults are applied.

## Primary Resolution

Preset run primary assignment priority:

```text
preset explicit node assignment
> preset stage-type default
> preset common default
> global stage-type default
> global common default
> fail fast
```

```toml
[default_stage_agents]
agents = ["codex"]

[default_stage_agents.stage_types.review]
agents = ["claude", "antigravity"]
```

Rules:

- Preset defaults are local to the preset and override global defaults.
- Explicit node assignments override preset and global defaults.
- Default lists may contain at most `MAX_FANOUT_AGENTS = 6` agents.
- `execute` must resolve to exactly one agent after all default inheritance.
- Assigned worker agents must exist and have the node stage capability.

Run creation writes the resolved result to `stage_assignments` and records
`assignment_provenance` using node ids.

## Failure Policy

Workflow TOML owns baseline flow semantics. Preset policy may keep or tighten
those semantics for one named run, but it cannot loosen workflow policy.

```toml
[failure_policy.stage_types.verify]
mode = "required"
max_fallback_agents = 2

[failure_policy.nodes.decide_2]
mode = "terminal"
```

`failure_policy.mode`:

| mode | meaning |
| --- | --- |
| `allow` | Fallback is allowed. If no fallback candidate resolves, the node fails and waits for intervention. |
| `required` | Run creation must resolve at least one fallback candidate. Runtime still respects `max_fallback_agents`. |
| `terminal` | Fallback is disabled. The node fails immediately when its lane cannot complete. |

Merge rules:

1. Compute workflow effective policy: system default -> workflow stage-type ->
   workflow node.
2. Apply preset policy: workflow effective -> preset stage-type -> preset node.
3. Policy fields inherit independently.
4. Strictness is `allow < required < terminal`; preset policy cannot move to a
   less strict mode than workflow effective policy.
5. If workflow explicitly set `max_fallback_agents`, preset may only keep or
   reduce that value.
6. `terminal` must not keep or inherit `max_fallback_agents`; the resolved packet
   stores only `{ "mode": "terminal" }`.

## Fallback Routing

Preset fallback routes override global fallback routes and may target stage
types or specific node ids.

```toml
[fallback]
agents = ["claude"]

[fallback.stage_types.verify]
agents = ["antigravity", "local-verifier"]
inherit_common = true

[fallback.nodes.verify_2]
agents = ["codex-reviewer"]
inherit_stage_type = true
inherit_common = false
```

Fallback candidate priority:

```text
preset node fallback
> preset stage-type fallback
> preset common fallback
> global stage-type fallback
> global common fallback
```

Rules:

- Each configured fallback agent list is limited to `3` agents.
- `inherit_common` and `inherit_stage_type` default to `true`.
- Node fallback supports both inheritance flags.
- Stage-type fallback supports `inherit_common`.
- Inheritance happens before de-duplication, primary-agent filtering, capability
  validation, and truncation by `failure_policy.max_fallback_agents`.
- Pure `current` nodes never resolve fallback candidates. `mode = "required"` on
  a pure `current` node is a creation error.

## Invocation And Timeout Resolution

Run creation materializes invocation lanes:

- primary lanes: `lane_id = "<node-id>:<agent>"`
- pure current lanes: `kind = "current"`, `timeout_seconds = null`
- plan/decide synthesis lanes, only when the node has multiple primary agents:
  `lane_id = "<node-id>:synthesis"`

Timeout priority:

```text
CLI --timeout-seconds
> preset fallback timeout (for fallback lanes)
> global fallback timeout (for fallback lanes)
> agent timeout_seconds
> system default 900
```

Timeouts must be `30-3600` seconds for spawned agent attempts. Current lanes use
`null`. `timeout_provenance` records `cli`, `preset_fallback`,
`global_fallback`, `agent`, `system_default`, or `current` per lane.

## Packet Output

Preset runs create current packet schema status with:

- `preset` and `preset_source` metadata.
- node-id keyed `stage_assignments`.
- node-id keyed `stage_invocations`.
- node-id keyed `stage_failure_policies`.
- node-id keyed `stage_fallbacks`, using empty agent lists for terminal/current
  nodes.
- empty node-id keyed `stage_attempts`, to be appended by dispatch/retry.
- `assignment_provenance`, `fallback_provenance`, and `timeout_provenance`.

Dispatch appends `stage_attempts` records with a monotonic `lane_attempt` for
the lane and a per-agent `attempt` counter for the actual agent. A fallback
agent with `max_attempts_per_agent = 2` can therefore record `attempt = 2` while
the lane continues to increment `lane_attempt` across primary and fallback
tries.

After packet creation, dispatch does not re-read the preset.
